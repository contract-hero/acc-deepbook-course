Flash-Loan Arbitrage with the Hot Potato

One PTB: borrow, use, repay — or it never happened

Borrow DEEP with no collateral, run an arb step, repay the exact principal — all inside a single transaction. The FlashLoan potato has no drop: it MUST die in the same tx, or everything reverts. That's what makes it risk-free for LPs.

1Connect & fund — a client and a BalanceManager

Same starting point as any DeepBook trade — client, funded signer — plus a BalanceManager used to top up the pool so the loan has DEEP to draw.

Key moment: a BalanceManager deposit alone doesn't fund the loan — borrow_flashloan_base draws from the pool vault's base_balance, which is separate custody. seedPoolBaseLiquidity escrows base into the vault by resting a POST_ONLY DEEP ask (the pool joins that base on settle), so the borrow never aborts with ENotEnoughBaseForLoan. A sandbox-determinism crutch; production pools hold standing liquidity and real integrations skip it.

2The hot potato — a value the compiler won't let you drop

borrow_flashloan_base hands back the borrowed coin and a FlashLoan with no abilities. Understand why that makes a collateral-free loan safe.

Key moment: FlashLoan has no drop, store, or key, so it can only be passed by value into pool::return_flashloan_base. If the PTB ends with it alive, the whole transaction aborts. That compile-time fact — repay-in-full-or-nothing-happened — is the entire safety model: the type system is the collateral.

3Consume it — execute_base merges, repays exactly, settles the rest

The function that discharges the potato: merge the loan into your working coin, repay the exact principal, transfer the remainder back.

Key moment: repay exactly the principal. Too little trips ERepayShort (your named guard, before the split); too much trips the vault's EIncorrectQuantityReturned — these loans carry no interest, so over-repaying aborts just as hard. split peels off precisely the principal; the remainder (your "profit") goes back to you.

4Publish — stage the package and write deployment.json

The TS driver reads the package id from deployment.json, so the module must be live on localnet before the test can run.

Key moment: stage the module inside the sandbox tree (so token/deepbook deps resolve) and ship no [addresses] block — a named-address module binds its address via the localnet environment at publish time; an explicit block makes the build reject localnet. The written arbExecutorPackageId is chain-specific — re-genesis ⇒ re-publish.

5Assemble the PTB — borrow and settle in one atomic transaction

Borrow with borrowBaseAsset and call execute_base in the same Transaction — that single-PTB constraint is the atomicity.

Key moment: the borrow thunk and the execute_base move call live in the SAME tx — there is no moment where the chain has lent you money and not been repaid. Split them across two transactions and the first aborts; the potato can't cross a tx boundary. Scale the borrow_amount u64 (DEEP 1e6) — the module repays the scaled value, not the human one.

6Drive it & prove atomicity — the live revert gate

Wire the core into the smoke driver and run the live suite — including the test that proves the loan unwinds completely when repayment falls short.

Key moment: asserting where the abort fired — inside execute_base (ERepayShort), NOT at the borrow step (ENotEnoughBaseForLoan, same generic code 1) — proves the borrow succeeded, the arb ran, and the short repayment reverted the entire settled PTB. The loan commits completely or unwinds completely; there is no halfway state where the lender is out their principal.
