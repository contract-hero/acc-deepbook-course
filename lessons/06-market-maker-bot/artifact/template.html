A two-sided grid resting around mid

One BalanceManager backs both sides. Bids below mid, asks above — every leg POST_ONLY so the maker never crosses the line.

1One BalanceManager for both sides

A maker holds inventory. Bids need quote coin (SUI), asks need base coin (DEEP) — both draw from a single shared on-chain BalanceManager.

Key moment: one shared BalanceManager is the inventory wallet behind every leg of the grid — deposit DEEP and SUI into it and all bids/asks settle there. Because createAndShareBalanceManager shares the object, you read its id out of effects.changedObjects (the freshly-Created object whose type includes BalanceManager), then re-register it on the client as MANAGER_1.

2Compute the grid around mid

Quote a bid and an ask at each level, symmetric around mid, so you earn the round-trip spread. First read mid, then lay out the ladder.

Key moment: read mid through withRetry (the sandbox's simulate endpoint blips on warmup), then for level i offset (spreadBps/10_000)·i on each side — bid below, ask above. round() floors every price to the pool's TICK; an unrounded price is rejected. Symmetry about mid is what makes it two-sided quoting.

3Rest the grid with POST_ONLY

A taker crosses the spread and pays. A maker rests on the book and collects. POST_ONLY is the on-chain enforcement of that distinction.

Key moment: orderType: OrderType.POST_ONLY tells the matching engine to reject any leg that would immediately cross and fill — so the bot can never accidentally become a fee-paying taker, it only ever rests liquidity. payWithDeep: false because the pool is whitelisted. Too-tight a spread? POST_ONLY does its job and you see fewer than four orders.

4Stake DEEP for fee rebates

A maker earns the spread. A staked maker earns the spread plus rebates. Stake DEEP through governance to qualify the account.

Key moment: governance.stake(poolKey, bmKey, amount) upgrades a plain maker into a rebate-earning one — staked DEEP qualifies the account for maker fee rebates that accrue in the same BM. A fresh stake lands in inactive_stake this epoch and rolls into active_stake next, so any correctness check must sum both.

5Unwind, re-quote, and run the gate

A real maker loops: cancel the whole book, re-read mid, re-quote. cancelAllOrders is the atomic unwind that makes that safe.

Key moment: cancelAllOrders tears down every resting leg atomically — the foundation of the cancel → midPrice → re-quote refresh loop (no half-cancelled book). The live gate proves the round trip via accountOpenOrders: exactly 4 after quoting, 0 after cancel-all. That, plus the stake check, is the lesson's final_verification.
